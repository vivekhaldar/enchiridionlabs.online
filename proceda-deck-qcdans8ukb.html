01  /  The shift

Harness engineering.

The bottleneck isn't what you say to the model — it's what you let the model do, when, and how.

For procedural tasks, a well-designed execution harness substitutes for raw model capability. Gemini 2.5 Flash, on rails, beats Claude 4.5 Opus off them.

Enchiridion LabsProceda 01 · the shift
02  /  Thesis
Architecture matters more than model strength — once the work is procedural.

SOPs have defined steps, tool dependencies, and deterministic outcomes. The model doesn't need to "be smart" — it needs to follow instructions and call the right tool with the right parameters.

Enchiridion LabsProceda 02 · thesis
03  /  Two architectures

Plan-and-execute vs. step-and-signal.

Baseline · ReAct / Function-Calling

Full 30-step SOP in the system prompt.

All tool schemas injected at once.

Model is planner and executor.

Patient Intake: 0% (Claude 3.5 Sonnet v2).

Proceda · State machine

SOP parsed into discrete SkillStep objects.

Each step receives only its instructions.

Model calls tools, then complete_step().

Patient Intake: 97% (Gemini 2.5 Flash).

Enchiridion LabsProceda 03 · two architectures
04  /  The pipeline

Convert. Execute. Extract.

Phase 1

Convert

LLM transforms unstructured SOP text into a structured SKILL.md. Zero hand-editing.

→
Phase 2

Execute

Step-by-step state machine. Model sees only the current step. Control tools gate progress; circuit breakers catch loops.

→
Phase 3

Extract

Output fields declared up front. Model emits XML tags. Deterministic parse — no regex on prose.

Decision 01  /  State machine

Reduce the reasoning surface per LLM call.

One step at a time. The model never sees the whole plan.

The executor walks the SOP as a loop. Each iteration: prompt the LLM with one step, run the inner loop until complete_step fires, advance.

src/proceda/internal/executor.pyL89–L122 View source → 
while session.current_step <= self._skill.step_count:
    step = self._skill.get_step(session.current_step)

    await self._emit(
        RunEvent.create(
            session.id,
            EventType.STEP_STARTED,
            {"step_index": step.index,
             "step_title": step.title},
        )
    )

    # Pre-approval gate (Decision 4)
    if step.requires_pre_approval:
        decision = await self._request_approval(...)
        ...

    # Run the inner LLM loop until complete_step()
    step_result = await self._execute_step(step.index)

    session.complete_current_step()
    session.advance_step()

Patient Intake · 6-tool dependency chain
97%

with Gemini 2.5 Flash on a domain where baseline ReAct/FC architectures score 0% — even with Claude 3.5 Sonnet v2.

Why the gap is categorical

  • The final tool needs outputs from all 5 prior tools.
  • Baselines must plan and execute the chain.
  • Proceda decomposes the chain into 6 isolated steps — each with one tool call.
  • Claude 4.1 Opus reaches 100% on baselines. Flash matches it on Proceda for a fraction of the cost.

Decision 02  /  Control tools

Make progress legible.

No implicit step detection. The model declares done.

Two tools are injected alongside every skill: complete_step and request_clarification. The model must call one to advance.

Decision 02  /  Fallbacks

Soft nudge.
Then hard cap.

If the model rambles instead of calling a tool, the harness intervenes. Two tiers: a textual nudge to remind it about complete_step, then a forced completion.

Result: 100% execution-completion rate across all 10 runnable SOP-Bench domains. Zero stalls.

        "Step %d force-completed after %d "

        step_index, text_only_count,
    )

        "system",

        "text-only responses.",
    ))

# Soft tier: nudge every N text-only turns
if text_only_count % self._max_text_before_prompt == 0:
    session.add_message(RunMessage.create(
        "user",
        "You seem stuck. Call `complete_step` "
        "if done, or use a tool to make progress.",
    ))

Decision 03  /  Forgetting

Context is a resource — not an accumulator.

Preserve what matters. Drop what's old. Tell the model when you did.
  • System messages: always kept.
  • Critical messages (step prompts, clarifications): always kept.
  • The rest: fill remaining budget, newest first.
  • Insert a truncation notice so the model knows.
src/proceda/internal/context.pyL83–L130 View source → 
critical = [m for m in non_system if m.is_critical]
non_critical = [m for m in non_system if not m.is_critical]

system_tokens = sum(self.count_message_tokens(m)
                    for m in system_messages)
critical_tokens = sum(self.count_message_tokens(m)
                      for m in critical)
remaining_budget = budget - system_tokens - critical_tokens

# Fill remaining budget, newest first
kept_non_critical = []
tokens_used = 0
for msg in reversed(non_critical):
    msg_tokens = self.count_message_tokens(msg)
    if tokens_used + msg_tokens > remaining_budget:
        break
    kept_non_critical.append(msg)
    tokens_used += msg_tokens

if trimmed > 0:

Decision 04  /  Approval gates

Policy belongs in the harness, not the model.

A marker in the SOP becomes a structural pause — not a polite ask.

Two markers in SKILL.md: [APPROVAL REQUIRED] (post-step) and [PRE-APPROVAL REQUIRED] (pre-step). The parser detects them; the executor enforces them.

src/proceda/skills/parser.py + executor.py View source → 
# parser.py — detect markers in step content
_MARKER_PATTERNS = {
    StepMarker.PRE_APPROVAL_REQUIRED:
        re.compile(r"\[PRE-APPROVAL\s+REQUIRED\]", re.IGNORECASE),
    StepMarker.APPROVAL_REQUIRED:
        re.compile(r"\[APPROVAL\s+REQUIRED\]", re.IGNORECASE),
    StepMarker.OPTIONAL:
        re.compile(r"\[OPTIONAL\]", re.IGNORECASE),
}

# executor.py — enforce gate before step runs
if step.requires_pre_approval:
    decision = await self._request_approval(
        step.index, step.title, "pre_step",
        f"Step {step.index}: {step.title}\n\n{step.content}",
    )
    if decision == ApprovalDecision.REJECT:
        raise ApprovalRejectedError(
            f"Pre-approval rejected for step {step.index}")

Decision 05  /  Structured output

Structure the interface between model and downstream.

Declare fields up front. Force XML tags. Parse deterministically.

The prompt tells the model exactly what to emit; the extractor doesn't guess. No regex on prose.

if skill.output_fields:
    fields = ", ".join(skill.output_fields)

# output_extractor.py — how it is parsed
def _extract_xml_tags(content, expected_set):
    output = {}
    for col in expected_set:
        pattern = rf"<{col}>(.*?)</{col}>"
        match = re.search(pattern, content, re.DOTALL)
        if match:
            output[col] = match.group(1).strip()
    return output

Decision 06  /  Guard rails

Bounded execution is a feature.

Constants you can read. Failure modes you can name.
  • Text-only soft nudge — every 5 responses.
  • Text-only hard cap — force-complete after 15.
  • Total iterations per step — 50 max.
  • App tool calls per step — 20 max, then error-recovery (retry/skip/cancel).

Anatomy  /  The SKILL.md

An SOP is a markdown file.

YAML frontmatter declares the skill, its required tools, and optional output fields. Numbered ### Step N headings define the procedure. Markers gate approval.

No DSL. No graph builder. Just a file an operator can read.

Anatomy  /  The system prompt

What the model is told, once.

Every run opens with this preamble. It names the skill, the control tools, the rules around clarifications and tools, and — when relevant — the output-field contract.

Notice: the prompt is short, declarative, and infrastructural. The model isn't asked to plan. It's told the shape of the game.

src/proceda/llm/prompts.pyL9–L40 View source → 
You are executing the skill: {skill.name}
Description: {skill.description}

You must follow the steps in order. For each step:
1. Read the step instructions carefully
2. Use the available tools to accomplish the step's goals
3. Call `complete_step` with a summary when the step is done
4. If you need clarification, call `request_clarification`

Important rules:
- Complete one step at a time
- Do not skip steps unless they are marked [OPTIONAL]
- Do not proceed to the next step until you call `complete_step`
- If a step asks you to ask the user something, you MUST call
  `request_clarification`. Printing the question as text does
  NOT pause for input. Only `request_clarification` reaches the user.
- If a step instructs you to call a specific tool, you MUST call it.
- If you determine the final answer is already known, call
  `skip_remaining_steps` — this ends the procedure immediately.

Findings  /  The paradox

Thinking models lose on rails.

Referral Abuse v2 — same skill, three Gemini models. The one with extended reasoning trails the ones without.

ModelTypeTSR
Gemini 2.5 Pro thinking (extended reasoning) 74.5%
Gemini 3 Flash non-thinking, cheap 88.5%
Gemini 3.1 Pro non-thinking, mid-tier 99.0%

Thinking models spend compute deciding what to do. When the harness already specifies what to do, the extra reasoning is wasted — and sometimes wrong.

05  /  Takeaways

Five things to remember.

  • Harness > model. For procedural work, what the model is allowed to do matters more than which model it is.
  • Decompose, then constrain. One step at a time, with explicit progress signals and circuit breakers.
  • Forget on purpose. Context is a budget; trim newest-first with critical-message preservation.
  • Structure the output. XML tags + deterministic parse. The 30→81% jump was a prompt change, not a model change.
  • Instrument or you're guessing. Every breakthrough came out of the JSONL traces.
